11)learn how to manage on hand quantity for products in inventory'
Perfect topic — on-hand quantity is the backbone of Odoo Inventory. Once this clicks, 80% of inventory confusion disappears. Let’s go step by step and keep it practical.
1. What “On-Hand Quantity” actually means
On-Hand Quantity =
Stock that physically exists right now in your internal locations
It:
✅ Increases when you receive products
✅ Decreases when you deliver products
❌ Does NOT include future receipts
❌ Does NOT include future deliveries
Think of it as “what’s on the shelf at this moment.”

2. Where to see On-Hand Quantity
Product level
Inventory → Products → select product
You’ll see:
On Hand
Forecasted
Reserved
💡 If On-Hand ≠ reality → there’s a process issue, not a math issue.

3.3. How On-Hand Quantity changes (this is key)
A. Incoming stock (increase)
Vendor Receipts
Inventory Adjustments (manual)
Manufacturing completion
Returns from customers
Rule: Stock increases only when the operation is Validated

B. Outgoing stock (decrease)C. Internal transfers (no net change)
Moving between locations
Warehouse transfers
Total On-Hand stays the same, location stock changes.
Customer Deliveries
Scrap
Returns to vendors
Manufacturing consumption
Rule: Draft or waiting operations do nothing.

4. Setting initial On-Hand Quantity (first time setup)
Correct way
Inventory → Operations → Inventory Adjustments
Create new adjustment
Select product & location
Enter counted quantity
Apply
⚠️ Never create fake purchase orders just to load stock.

5. Inventory Adjustments (ongoing corrections)
Use adjustments when:
Physical count differs from system
Stock loss or damage
Initial migration
Annual stock count
Best practice:
Do cycle counts (small, frequent)
Limit who can apply adjustments
Always document the reason

6. Forecasted vs On-Hand (common confusion)
Quantity Type	Meaning
On-Hand	Physically available now
Reserved	Allocated to orders
Forecasted	On-Hand + Incoming − Outgoing

💡 You can sell based on Forecasted, but ship based on On-Hand.

7. Reservations & their impact
When a Sales Order is confirmed:
Stock may be Reserved
On-Hand stays the same
Available stock decreases
Example:
On-Hand: 50
Reserved: 20
Available: 30
This prevents double-selling.

8. Multi-location & warehouse impact
If you use multiple locations:
On-Hand is sum of all internal locations
Each location has its own quantity
Check via:
Inventory → Reporting → Inventory Report
This avoids the “we have stock but not here” problem.

9. Reordering Rules (protect On-Hand)
Set minimum stock levels:
Product → Reordering Rules
Define Min / Max
Odoo will:
Alert you
Auto-create RFQs or MOs
This keeps On-Hand from hitting zero unexpectedly

10. Common mistakes (and how to avoid them)
🚫 Editing stock directly on the product
→ ✅ Always use Inventory Adjustments
🚫 Forgetting to validate receipts/deliveries
→ ✅ No validation = no stock change
🚫 Using Consumable instead of Storable
→ ✅ Consumables don’t track On-Hand properly
🚫 Allowing too many users to adjust stock
→ ✅ Restrict permissions

11. Best daily habits for clean On-Hand quantities
Validate stock moves the same day
Count high-value items more often
Investigate negative stock immediately
Use barcodes if volume is high
Align warehouse & accounting processes

12. Quick real-world example
You have
On-Hand: 100 units
Confirm Sales Order for 30
→ On-Hand = 100
→ Reserved = 30
→ Available = 70
Deliver 30 units:
→ On-Hand = 70
→ Reserved = 0
Clean. Predictable. Auditable.

90)explore varient creation models and understand how dynamic varient options work
Great topic — product variants are one of Odoo’s most powerful (and most misunderstood) features. Let’s break this into variant creation models first, then zoom in on dynamic variant options so the logic really clicks.

1. What is a Product Variant in Odoo?

A product template = the generic product
A product variant = a specific combination of attributes

Example:

Product: T-Shirt

Attributes: Color, Size

Variants:

T-Shirt / Red / M

T-Shirt / Blue / L

Each variant has:

Its own SKU

Its own on-hand quantity

Its own price (optional)

2. Variant creation models (this is the foundation)

Odoo supports three variant creation modes per attribute.

Go to:
Inventory or Sales → Configuration → Attributes

A. Instantly (Classic model)

How it works

All possible combinations are created immediately

Example:

Color (3) × Size (4) = 12 variants

Best for

Limited combinations

Apparel basics

Items where every combo is sellable

Impact

Variants exist in database

Stock can be managed immediately

Fast sales experience

⚠️ Risk: Variant explosion if attributes grow.

B. Dynamically (On-demand model)

How it works

Variants are created only when used

Happens when:

Sales Order is confirmed

Purchase Order is confirmed

Manufacturing Order is created

Best for

Custom products

Large attribute combinations

Low-frequency variants

Impact

Cleaner database

Lower maintenance

Slightly more complex logic

💡 This is where dynamic variant options shine.

C. Never (Informational model)

How it works

Attribute does NOT create variants

Used only for description

Example:

Engraving Text

Customer Name

Custom Message

Best for

Personalization

Non-stock attributes

Impact

No SKU split

No stock separation

Appears on documents only

3. Dynamic Variant Options — deep dive

Dynamic variants rely on attribute + value configuration.

How the dynamic process works

Product template is created

Attribute set to Dynamically

No variants exist initially

User selects attribute values on SO/PO

Odoo creates the exact variant needed

That variant is reused in the future

📌 Variants are cached, not recreated every time.

4. Attribute value controls (very important)

Each attribute value can control behavior.

Go to:
Attribute → Values

Options you can define:

Extra Price

Exclude for (variant exclusion rules)

Visible on eCommerce

Default value

Variant exclusion rules
Prevent invalid combinations.
Example:
Size XL ❌ Color Yellow
This applies to both instant and dynamic models.
5. Dynamic variants in Sales & eCommerce
Sales Orders
Attribute selection happens at order line
Only valid combinations selectable
Variant created on confirmation
eCommerce
Variant created when customer adds to cart
Invalid combinations are disabled automatically
Pricing updates live
This is why dynamic options feel “smart.”
6. Stock & accounting impact
Dynamic or instant — once created:
Variant has its own stock
Own cost
Own valuation
Own reordering rules
No difference after creation.
The difference is when the variant exists.
7. Real-world comparison
Example: Custom Laptop
Attributes:
RAM (8/16/32 GB) → Dynamic
Storage (512/1TB) → Dynamic
Keyboard Language → Never
Possible combinations: 12
Actual sold: maybe 3
Dynamic model wins here.

8. Common mistakes (and how to avoid them)

🚫 Setting everything to “Instantly”
→ Variant explosion

🚫 Using “Never” for stock-affecting attributes
→ Inventory chaos

🚫 Forgetting exclusion rules
→ Invalid orders

🚫 Mixing dynamic + instant without planning
→ Confusing behavior
9. Best practice decision guide
Scenario	Recommended Model
Apparel basics	Instantly
Configurable products	Dynamically
Engraving / text	Never
High-volume SKUs	Instantly
Low-frequency custom	Dynamically
10. Mental model (remember this)
Instant = “Create everything now”
Dynamic = “Create only when needed”
Never = “Describe, don’t split”
Once you think this way, variant behavior becomes predictable.

study how CRM links with existing contacts and how new contacts are created directly from CRMGreat question — this is a core CRM concept in Odoo, and once you understand it, lead/customer data stays clean instead of turning into a mess. Let’s study it logically, from structure → flow → best practices.

1. One key idea (everything depends on this)

CRM does NOT have its own contact database.

CRM uses Contacts (Partners) from the Contacts app.

So:

Leads

Opportunities

Customers

👉 All point to the same contact records

No duplication unless users create it.

2. How CRM links to existing contacts
A. Lead / Opportunity → Customer field

In CRM, every Lead or Opportunity has a Customer field.

That field:

Searches existing contacts

Links directly to a partner record

Does NOT create a new contact automatically

Example:

Lead created → Customer = ABC Corp

CRM links to the existing ABC Corp contact

All history stays in one place

B. What gets linked automatically

When a customer is linked:

Address

Email

Phone

Company info

Existing sales history

CRM simply reads from Contacts.

3. How new contacts are created from CRM

There are three main creation paths.

A. Manual creation from Lead / Opportunity

If the contact does not exist:

Open Lead / Opportunity

Type a new name in Customer field

Click Create or Create and Edit

Odoo creates:

A new Contact (Company or Individual)

Links it instantly to the CRM record

💡 This is the most common flow.

B. Automatic creation when converting Lead → Opportunity

When you click Convert to Opportunity:

Odoo can:

Link to existing contact

OR create a new contact automatically

Based on:

Email

Company name

Your selection in the conversion wizard

This prevents losing lead info.

C. Creation via incoming email / website / form

CRM can auto-create contacts when leads come from:

Website contact forms

Lead forms

Email aliases

What happens:

Lead is created

Contact is created (or matched) based on email

CRM links them together

⚠️ This is where duplicates can sneak in if not configured.

4. Company vs Individual creation logic

CRM decides based on:

Company name field

Email domain

User choice during conversion

Example:

john@acme.com

Company name = ACME

→ Company contact: ACME
→ Individual contact: John (child contact)

Best practice:

Always convert B2B leads into Company + Contact person

5. What data moves from CRM to Contacts

When a contact is created from CRM, Odoo transfers:

Name

Email

Phone

Mobile

Company name

Address (if provided)

After creation:

Contact becomes reusable across Sales, Invoicing, Inventory

CRM remains a process layer, not a data store.

6. What happens when a linked contact is edited

Contacts and CRM stay fully synced.

Edit in Contacts app:

CRM updates instantly

Edit in CRM:

Contact updates instantly

There is only one record, not two.

7. Duplicate prevention (very important)

Odoo uses:

Email matching

Company name matching

Manual user confirmation

Best practices:

Enable Lead Enrichment / Deduplication

Train users to search before creating

Use company as parent contact

Standardize naming rules

8. CRM → Sales → Accounting chain

Once linked:

Opportunity → Quotation

Quotation → Sales Order

Sales Order → Invoice

All use:
👉 the same Contact

This ensures:

Correct billing

Correct delivery

Correct accounting entries

9. Common mistakes (and how to avoid them)

🚫 Creating contacts manually before checking CRM
→ ✅ Always search first

🚫 Creating individuals instead of companies
→ ✅ Company first, people as children

🚫 Letting CRM auto-create unchecked contacts
→ ✅ Review leads before conversion

🚫 Editing contact info in multiple places
→ ✅ One source of truth: Contacts app

10. Simple mental model

Contacts = Master data

CRM = Sales process

Leads = Potential

Opportunities = Qualified potential

Customers = Contacts reused everywhere

CRM points to contacts — it does not replace them.
